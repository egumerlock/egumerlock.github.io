Ruby Hashes vs Javascript Objects

March 3rd, 2016
By Eric Gumerlock

Ruby Hashes vs Javascript Objects Although they might look extremely similar, Ruby and Javascript have noticeable syntax differences, especially when it comes to nitpicky notation such as the “;” that is necessary at the end of Javascript statements. While there a number of differences in Ruby and JS, I am focusing on explaining the difference between the Ruby hash and the JS Object. Lets start off with the similarities, as they do roughly the same thing - handle composite data structures that can include many different kinds of information (strings, classes, symbols, arrays, other hashes, etc. They look alike given by their literal notations: 

      # Ruby hash:
      book = {
        name: “A Wrinkle in Time”,
        year: 1963,
        length: 500,
        publisher: “Farrar, Straus & Giroux”
      }
      // JS object:
        var book = {
        name: “A Wrinkle in Time”,
        year: 1963,
        length: 500,
        publisher: “Farrar, Straus & Giroux”
      };
Nearly the same! Aside from a few more details required in the JS Object. Accessing their values is also similar, but actually has quite different syntax. 

      # Ruby hash:
      book[:year]
      => 1963
      // JS object:
      book.year;
      => 1963
Very similar so far, as both consist of a property and a value for that property. However, the assigner is referred to as a key, whereas in JS it is called a property, but in the end they both find the same value. The difference here is that in Ruby, keys can be any data type (symbols, strings, etc), but in JS the properties are simply standardized as that: properties of the object that can be called upon by their relative methods. Yet, the similarities keep popping up - both in Ruby hashes and JS objects the values can be nearly any data type (arrays, hashes, strings, numbers, booleans). One major difference is that properties in JS can be functions, whereas Ruby hash values cannot be the Ruby equivalent, methods. This is definitely an advantage for Javascript, but this doesn’t make Ruby lesser at all. We would code this into JS like so: 

    // JS Object
      book.read = function() {
        console.log(“You are about to read” + book.name)
         } ;
      book.read();
You can see in this example that book.name is referred to in the printing statement quite easily. Also, this entire function is saved as a property in the book object. As Ruby cannot save a method as a value, you would have to create the method as a separate entity from the hash. 

    # Ruby
    def read(book)
      puts “You are about to read #{book[:name]}”
    end

    read(book)
This code is disconnected from the hash altogether, and also the reference to book[:name] value is quite a bit harder to type that the one in JS. This is fixed in Ruby with classes, which gives the ability to reach values in a similar fashion to the one in JS. For quick and simple references to property values, I think JS wins on this one.
